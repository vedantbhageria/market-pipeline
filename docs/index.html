Market Pipeline

A real-time cryptocurrency market-data engine. It ingests live Binance USD-M futures trades over WebSocket, backfills recent history over REST, computes a suite of rolling technical indicators, stores everything in Redis Streams, and serves it to a live web dashboard.

Python Redis Streams asyncio + threading + multiprocessing FastAPI / uvicorn WebSockets python-binance

The idea

One launcher (pipeline.py) spins up four independent services, each in its own console window. They never call each other directly — they communicate only through Redis Streams (data) and Redis Pub/Sub (signals).

Why decoupled

Each service can crash, restart, or scale on its own. The metric workers and dashboard simply read whatever is in Redis; producers (live + backfill) simply write. No service holds a reference to another.

Architecture

Four services orchestrated by a process manager, all meeting at Redis.

Data flow

1. Cold start & the backfill handshake

Because live ticks and historical bars both land in the same LiveTicker:{symbol} stream, ordering matters. Live ingestion connects immediately but buffers every incoming tick per symbol. Backfill works through history; when a symbol's history is stored, it publishes that symbol on the backfill:complete channel. Live is subscribed — on receiving the signal it flushes that symbol's buffer (in order) and switches to writing ticks directly from then on.

# per symbol, inside Live ingestion tick arrives ─▶ is backfill done for this symbol? ├─ no ─▶ append to in-memory buffer └─ yes ─▶ first time? flush buffer in order, then xadd directly to LiveTicker:{symbol}

2. Metrics

Each metric worker xreads the tick streams, maintains a rolling time window (METRICS_WINDOW_MS = 30s) as a deque per symbol, runs an incremental compute function as each tick enters/leaves the window, and xadds the result to that metric's own stream — keyed with the same entry id as the source tick so outputs align 1:1 with inputs.

3. Serving

The dashboard opens a WebSocket per browser. When a client subscribes to a symbol, the server starts a single tail task for that symbol that reads the tick + all metric streams and fans the data out to every client watching it. History (quick + full-window) is loaded via xrevrange on connect.

Orchestration

pipeline.py is the only thing you run. It creates a ProcessManager, starts the connection-registry monitor, then launches each service.

pm = ProcessManager()
pm.start_monitor()          # background thread: maintains the Connections hash
pm.start("Live Binance")
pm.start("Backfill")
pm.start("Server")          # uvicorn on :8000
time.sleep(2); webbrowser.open("http://localhost:8000")
pm.start("Metrics")         # run_all.py → 8 workers
# Ctrl+C → pm.stop_all()

ProcessManager

start(name)Looks the launch command up in the registry and spawns it. On Windows each gets a new console window (CREATE_NEW_CONSOLE). stop(name)Kills the whole process tree — taskkill /F /T on Windows, terminate() elsewhere. stop_all()Stops every tracked process (the Ctrl+C path). start_monitor()Starts the ManageAllConnections daemon thread.
Every child is launched with cwd = project root and the root injected into PYTHONPATH, so import services / import shared resolve regardless of how the child is started or the editable-install state.

Live ingestion services/data/live_binance.py

Connects to Binance USD-M futures combined streams and consumes aggTrade events.

  • Subscribes to {symbol}@aggTrade for the whole universe, split into shards of LIVE_WS_SHARD_SIZE (190) streams per WebSocket connection.
  • Each shard runs as its own auto-reconnecting asyncio task.
  • A separate task listens on the backfill:complete Pub/Sub channel and sets a per-symbol asyncio.Event.
  • Buffer-then-flush: ticks are held until that symbol's event fires, then the buffer is flushed in order and subsequent ticks are written directly.

Tick written to LiveTicker:{symbol}

id     = "{T}-{a}"            # trade time (ms) + aggregate-trade id
fields = { price, quantity }
maxlen = STREAM_MAXLEN (36000, approximate)
Using {T}-{a} as the Redis entry id makes ingestion idempotent: re-delivering the same trade can't create a duplicate stream entry.

Backfill services/data/backfill_binance.py

Fetches recent aggregate-trade history over the REST API and lays it down as 1-second bars so the live stream has immediate context on startup.

Resume logic

On start it calls latest_key(...) to find, per symbol, the last stored tick. If that tick is recent (within BACKFILL_LOOKBACK_MS) it resumes from there; otherwise it starts from the default lookback window. This means restarts don't refetch what's already stored.

Rate limiting — RequestQueueManager

A token-bucket limiter shared across BACKFILL_NUM_WORKERS (6) request queues and worker threads.

capacityburst budget (tokens), here 100/sec refill_ratetokens added per second = budget_per_min / 60 (2200/min) weighteach aggregate-trades call costs BACKFILL_AGG_TRADE_WEIGHT = 20 report_rate_limit()on a Binance -1003 / ban, zeroes tokens and pauses all workers until the ban timestamp (parsed from the error) or 15s

Aggregation

aggregate_to_seconds() buckets raw trades by their second, keeping the last price and summing quantity, producing one bar per second written with id {open_time}-0. After storing a symbol's bars it publishes the symbol on backfill:complete.

Metric workers services/metrics/

All eight indicators share one engine, RollingMetricWorker. A worker is configured with a source stream, an output stream, a window size, and a pure compute function.

The rolling engine

  • Sharded across processes: each metric fans its symbols across num_workers (4) OS processes via multiprocessing.
  • Per-symbol state: a deque window of (ts, price, qty), a running quantity sum, and the previous result.
  • Resume: resume_cursors() rewinds each cursor to window_ms before the last output already written, so the 30-second window is rebuilt from existing history on restart.
  • Incremental: on each tick, expired entries are popped and the compute fn receives (window, popped, appended, prev, qty_sum) — letting it update in O(1) instead of rescanning.

compute function contract

def metric(window, popped, appended, prev, qty_sum):
    # window   : entries currently in the window (excl. the new one)
    # popped   : entries that just expired this tick
    # appended : the new (ts, price, qty)
    # prev     : the previous result for this symbol (or None)
    # qty_sum  : running sum of quantity over the window
    return value            # or a dict of fields

run_all.py imports all eight pre-built workers and runs each in its own thread; inside, every worker spawns its own pool of processes. So one "Metrics" console window hosts the lot.

Dashboard server services/dashboard_server/

A FastAPI app (served by uvicorn) that streams data to the browser over WebSocket and exposes REST history.

WebSocket protocol /ws

The browser sends JSON actions; the server pushes typed frames back.

Client → serverEffect
{action:"subscribe", symbols:[…], window}Subscribe (max 12 symbols/client). Sends quick history immediately, then a full-window backfill.
{action:"set_window", symbol, window}Change the history window for a symbol (2m / 10m / 30m / 1h).
{action:"unsubscribe", symbols:[…]}Stop receiving a symbol.

Fan-out — SubscriptionManager

Tracks the client↔symbol mapping both ways. The first client to watch a symbol starts a single _tail_symbol task; the last to leave cancels it. That one task reads the tick stream + every metric overlay for the symbol and broadcasts to all its watchers — so N clients on the same symbol cost one Redis reader, not N.

Connection registry shared/redis_pool.py

Every process creates its own ConnectionManager; Redis client objects are not shared across processes. The registry exists so you can observe all connections across all processes from one place.

Per-process side

Each named get_sync / get_async creates a client once, caches it, and publishes an add event on the Connections Pub/Sub channel. Construction also registers a {name}-managerial client.

Monitor side

ManageAllConnections() (one daemon thread, started by the ProcessManager) subscribes to that channel and rebuilds a Redis hash named Connections: manager → JSON list of [connection, type].

HGETALL Connections
  "live"      → [["live-managerial","sync"], ["live-async","async"]]
  "backfill"  → [["backfill-managerial","sync"], ["backfill-sync","sync"]]
  "dashboard" → [["dashboard-managerial","sync"], ["dashboard-async","async"], …]
  "Monitor"   → [["RedisConnectionMonitor","sync"]]
Configurable host/port. All clients read REDIS_HOST / REDIS_PORT from the environment, defaulting to localhost:6379.

Setup & running

Prerequisites

  • Python 3.10+
  • A running Redis server (default localhost:6379)
  • Dependencies: redis, websockets, python-binance, fastapi, uvicorn, numpy, psutil

Install

# from the project root — registers the packages for clean imports
pip install -e .

Run the whole pipeline

python pipeline.py

This opens separate console windows for Live, Backfill, Server and Metrics, then opens the dashboard at http://localhost:8000. Press Ctrl+C in the launcher to stop everything.

Run a single service

python services/data/live_binance.py
python services/data/backfill_binance.py
python services/metrics/run_all.py
python -m uvicorn services.dashboard_server.server:app --port 8000

Reset Redis

python scripts/flush_redis.py     # flushes ALL keys — dev only
Heads-up. flush_redis.py calls FLUSHALL — it wipes the entire Redis instance, not just this project's keys. Don't run it against a shared Redis.
